# 11_后端平台数据层与AI基础设施实施清单

## 目标

搭建 `AI 任务管家` 的后端基础平台，但以 **Windows 本地可运行** 为第一优先级，先满足“本地能开发、本地能验证、本地能跑通”的要求，再为后续升级到完整生产级基础设施预留路径。

---

## 交付范围

本目标覆盖：

- 服务端模块骨架
- 数据库与迁移
- 本地任务调度与异步任务承接
- 本地文件存储
- 检索与知识增强的可升级预留
- AI 网关
- 共享契约层

本目标不直接覆盖：

- 单个业务模块的完整交互
- 前端页面表现

---

## 主要实施项

### 1. 建立后端模块化单体

建议模块包括：

- `auth`
- `workspace`
- `goal`
- `planning`
- `execution`
- `knowledge`
- `collaboration`
- `export`
- `integration`

### 2. 建立数据库层

- 当前阶段使用 `SQLite`
- 使用 `Prisma`
- 建立 migration 流程
- 建立种子数据机制

说明：

- `SQLite` 更适合当前 Windows 本地环境
- 后续如果进入部署或多人协作阶段，再迁移到 `PostgreSQL`

### 3. 建立本地任务调度与异步承接层

- 使用数据库任务表承接异步任务
- 使用应用内轮询或轻量定时任务处理
- 任务重试策略
- 失败任务回收策略

说明：

- 当前阶段不强依赖 `Redis` 与 `BullMQ`
- 先用 `TaskJob` 一类的数据表配合应用内 worker 跑通导出、重规划、通知等任务
- 后续如果并发量提高，再升级到 `Redis + BullMQ`

### 4. 建立本地文件存储层

- 本地 `uploads/` 目录
- 导出文件
- 附件
- 图片
- 共享页静态资源

说明：

- 当前阶段先以本地目录存储为主
- 后续再迁移到对象存储服务

### 5. 建立 AI 基础设施

- 模型网关
- Prompt Registry
- 可插拔的知识增强入口
- 请求日志
- 成本统计

### 6. 建立检索与知识增强的升级预留层

- 当前阶段先支持关键词检索或轻量标签检索
- 预留向量写入与语义召回接口
- 后续再接 `pgvector` 或独立向量库

### 7. 建立共享契约层

- DTO
- 枚举
- 错误对象
- 异步任务状态对象

---

## 技术栈要求

- `NestJS`
- `TypeScript`
- `Prisma`
- `SQLite`（当前 Windows 本地可运行方案）
- 本地文件存储
- 数据库任务表 + 应用内轮询 / 定时任务
- 关键词检索或轻量标签检索

## 技术栈升级路径

当后续需要部署、协作或更高并发时，再升级为：

- `PostgreSQL`
- `Redis`
- `BullMQ`
- 对象存储
- `pgvector` 或独立向量库

---

## 架构重点

- 先采用模块化单体
- 后期热点模块再拆服务
- AI 调用必须经过服务端网关
- 数据层必须支持版本与审计
- 当前阶段所有基础设施选择都优先服从“Windows 本地可运行”

## 当前阶段推荐实现

如果只以“本地能跑”为目标，推荐落地组合为：

- 服务端：`NestJS`
- ORM：`Prisma`
- 数据库：`SQLite`
- 文件：本地 `uploads/`
- 异步任务：数据库任务表 + 应用内 worker
- 检索：关键词 / 标签检索
- AI：先接单模型或 mock，再逐步扩展

## 后续升级方向

当本地版稳定后，再逐步升级：

1. `SQLite` -> `PostgreSQL`
2. 数据库任务表 -> `Redis + BullMQ`
3. 本地文件目录 -> 对象存储
4. 关键词检索 -> 向量检索
5. 单机运行 -> 可部署多环境架构

## 完成标准

- 后端基础服务可运行
- 数据库迁移机制可用
- 本地异步任务机制可运行
- 本地文件存储可用
- AI 网关可统一接入模型
- 检索层至少可承接关键词或标签级调用
- 不依赖 `Docker`、`WSL2`、`Redis` 也能在 Windows 本地启动

---

## 建议拆分顺序

1. 先搭服务端骨架
2. 再接数据库
3. 再做数据库任务表与应用内 worker
4. 再接本地文件存储
5. 再做 AI 网关与轻量检索
6. 最后完善共享契约层
