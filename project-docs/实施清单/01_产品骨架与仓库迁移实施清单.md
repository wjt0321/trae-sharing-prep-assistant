# 01_产品骨架与仓库迁移实施清单

## 目标

把当前 `分享会筹备助手` 的比赛版单页 Demo，升级为可承接长期演进的在线产品骨架，为后续 `AI 任务管家` 的所有能力提供稳定载体。

---

## 交付范围

本目标覆盖：

- 新产品仓库或新目录结构的建立
- 前后端工程骨架初始化
- 设计系统与业务模块的基础分层
- 从当前 Demo 到新产品架构的迁移策略
- 首个场景的兼容承接方式

本目标不直接覆盖：

- 真实账号系统细节
- 完整 AI 规划逻辑
- 模板库、协作、自动化等业务功能

---

## 为什么先做

如果没有产品骨架，后续所有能力都会继续堆在当前 Demo 上，导致：

- 架构难以扩展
- 新旧文档和代码逻辑分裂
- 前后端边界不清
- 后续多场景与多角色能力难以落位

---

## 主要实施项

### 1. 明确代码组织策略

- 决定是新建长期产品仓库，还是在当前仓库中新增 `apps/web`、`apps/server` 之类的目录
- 保留当前 Demo 作为 `legacy-demo` 或 `showcase-demo`
- 确保未来母产品和比赛版原型能并行维护

### 2. 初始化前端工程骨架

- 使用 `Next.js + React + TypeScript`
- 建立 `app` 路由和基础页面目录
- 接入 `Tailwind CSS`
- 预留 `shadcn/ui` 组件层
- 建立全局 token、布局组件和状态容器

### 3. 初始化后端工程骨架

- 使用 `NestJS + TypeScript`
- 建立模块化单体结构
- 预留 `auth / goal / planning / execution / knowledge / collaboration / export / integration` 模块
- 接入 `Prisma`
- 连接 `PostgreSQL`

### 4. 建立共享契约层

- 定义前后端共享 DTO
- 统一错误码和响应结构
- 建立基础 API 约定
- 预留 AI 任务状态与异步任务状态枚举

### 5. 设计迁移路径

- 识别当前 Demo 中需要继承的内容：
  - 输入结构
  - 阶段结果展示方式
  - 行动清单收口
  - 当前视觉语言
- 定义哪些内容直接迁移，哪些内容只保留为参考

---

## 前端设计重点

- 首页与应用工作台必须分离
- 保留当前“结构化输入 -> 结果输出”的强闭环体验
- 新骨架必须支持多页面，不再局限单页
- 预留共享页、模板页和设置页入口

---

## 后端设计重点

- 服务结构必须支持后续模块拆分
- 先用模块化单体，避免过早微服务化
- 统一鉴权、日志、队列和数据库接入方式

---

## 数据与基础设施重点

- 初始化数据库迁移机制
- 初始化环境变量管理
- 初始化对象存储与缓存连接方式
- 预留异步任务队列

- 新产品工程可以本地启动
- 前后端工程目录稳定
- 已形成清晰的模块边界
- 当前 Demo 与未来产品骨架的关系已写清

1. 先定仓库组织方式
2. 再搭前端工程骨架
3. 再搭后端工程骨架
4. 再建立共享契约层
5. 最后梳理 Demo 迁移关系
