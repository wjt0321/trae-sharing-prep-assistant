## 阶段一：安全加固（P0 — 比赛提交前必须完成）

> 目标：消除 4 个 Critical 安全问题，达到可安全提交的比赛质量。

### 1.1 登录暴力破解防护

- [x]**邮箱维度失败计数**：在 `auth.service.ts` 的 `login()` 中增加失败计数逻辑（内存级 `Map<email, {count, lastAttempt}>`）
- [x]**递增延迟**：连续失败后逐次增加等待时间（第 5 次起 +5s，第 10 次起锁定 15 分钟）
- [x]**失败日志**：记录登录失败事件到审计日志（`AUDIT_LOGIN_FAILED`）
- [x]**前端反馈**：登录页展示剩余尝试次数或锁定提示

### 1.2 Refresh Token 安全存储

- [x]**Schema 变更**：`Session.refreshToken` 改为 `refreshTokenHash`（存储 SHA-256）
- [x]**签发逻辑**：`auth.service.ts` 的 `issueTokens()` 生成 refresh token → 计算 hash → 存 hash 到 DB → 返回原始 token 给客户端
- [x]**刷新逻辑**：验证时对传入 token 做 SHA-256，与 DB 中的 hash 比对
- [x]**过期时间统一**：DB `expiresAt` 从 `JWT_REFRESH_EXPIRES_IN` 配置动态读取，不再写死 7 天
- [x]**数据迁移**：`prisma migrate` 添加新字段，旧 `refreshToken` 字段标记废弃后删除
- [x]**向后兼容**：旧 session（有 `refreshToken` 无 `refreshTokenHash`）首次刷新时自动迁移到 hash 模式

### 1.3 AI 加密密钥独立化

- [x]**启动校验**：`ai-gateway.service.ts` 和 `ai-config.service.ts` 在启动时检查 `AI_CONFIG_ENCRYPTION_KEY`，未配置时拒绝启动或禁用 AI 功能（不再回退到 `JWT_SECRET`）
- [x]**环境变量**：`.env.example` 添加 `AI_CONFIG_ENCRYPTION_KEY` 说明
- [x]**解密失败处理**：若已存储的 Key 用旧密钥（JWT_SECRET）加密，提供迁移脚本或返回明确错误提示用户重新配置
- [x]**密钥轮换支持**：`crypto.util.ts` 增加 `reEncrypt()` 方法，支持用新密钥重新加密所有已存储的 API Key

### 1.4 Token 存储迁移到 httpOnly Cookie

- [x]**后端改动**：登录/刷新接口在响应中 `Set-Cookie`（`httpOnly` + `Secure` + `SameSite=Strict` + `Path=/api`）
- [x]**后端改动**：添加 `CookieAuthGuard` 或修改 `JwtAuthGuard` 同时支持从 Cookie 和 Authorization header 读取 token
- [x]**前端改动**：`api.ts` 移除 `tokenStorage` 的 localStorage 读写，`fetch` 添加 `credentials: 'include'`
- [x]**前端改动**：`auth.tsx` 的 AuthContext 不再从 localStorage 读取 token，改为从 `/api/auth/me` 获取用户状态
- [x]**CSRF 保护**：添加 `X-CSRF-Token` header 或使用 `SameSite=Strict` + 检查 `Origin`/`Referer` header
- [x]**登出接口**：`POST /api/auth/logout` 返回 `Set-Cookie` 清空 token cookie
- [x]**向后兼容**：过渡期同时支持 Cookie 和 Authorization header，前端优先用 Cookie

## 阶段二：架构与质量提升（P1 — 比赛后可迭代）

> 目标：消除竞态条件、N+1 查询、数据完整性风险，打好上线基础。

### 2.1 竞态条件修复

- [x]**syncFromPlan 修复**：`execution.service.ts` 的 `syncFromPlan()` 改用逐个 `create` + `$transaction([])` 替代 `createMany` + 反查，确保 ID 一致性
- [x]**ApiProviderConfig 唯一激活**：`schema.prisma` 的 `ApiProviderConfig` 添加应用层唯一性校验（`create`/`update` 时在事务中先取消其他 `isActive`）
- [x]**PromptTemplate 唯一激活**：`prompt-registry.service.ts` 的 `activate()` 确保同一 name 只有一个 `isActive=true`

### 2.2 批量操作与 N+1 优化

- [x]**batchUpdateStatus 优化**：前置批量 goal 成员校验 → 单个 `$transaction` 内完成所有任务更新 + 状态历史写入
- [x]**collaboration findAllComments 分页**：添加分页参数（`page`/`pageSize`），避免一次性加载所有评论
- [x]**execution page 历史查询合并**：后端新增 `/goals/:id/tasks/history/recent` 批量接口，前端从 20 个独立请求改为 1 个

### 2.3 数据完整性加固

- [x]**ExecutionTask 软删除**：添加 `deletedAt DateTime?` 字段 + `@@index([deletedAt])`
- [x]**Comment 软删除**：添加 `deletedAt DateTime?` 字段
- [x]**ActivityEvent 软删除**：添加 `deletedAt DateTime?` 字段
- [x]**TaskStatusHistory 软删除**：添加 `deletedAt DateTime?` 字段
- [x]**Assignment 软删除**：添加 `deletedAt DateTime?` 字段
- [x]**所有查询过滤**：相关 service 的查询增加 `deletedAt: null` 条件
- [x]**数据库迁移**：`prisma migrate` 添加字段，`seed.ts` 不生成软删除数据

### 2.4 数据库索引补充

- [x]`Goal.deletedAt` — `@@index([deletedAt])`
- [x]`ExecutionTask` — `@@index([goalId, status])` 复合索引
- [x]`ActivityEvent` — `@@index([goalId, createdAt])` 复合索引
- [x]`Comment` — `@@index([goalId, parentId])` 复合索引
- [x]`TaskJob` — `@@index([createdAt])`
- [x]`AiCallLog` — `@@index([createdAt])`
- [x]迁移后验证：使用 `EXPLAIN QUERY PLAN` 确认新索引被使用

### 2.5 前端错误边界与组件拆分

- [x]**execution/page.tsx**：提取 `TaskBoard`、`StatusModal`、`AssignModal`、`HistoryPanel`、`MilestonePanel` 到独立文件
- [x]**admin/page.tsx**：提取 `OverviewTab`、`BusinessTab`、`AuditTab` 到独立文件
- [x]**Error Boundary**：在 `app/goals/[id]/execution/` 和 `app/admin/` 添加 `error.tsx`（Next.js Error Boundary）
- [x]**共享状态样式**：提取 `STATUS_STYLE_MAP` 到 `lib/constants.ts`，消除 `TaskRow` 和 `StatusBadge` 中的重复

### 2.6 安全加固补充

- [x]**IntegrationController 认证**：添加 `@UseGuards(JwtAuthGuard)` 类级守卫
- [x]**生产环境异常过滤**：`global-exception.filter.ts` 的非 `HttpException` 分支在生产环境返回通用错误消息
- [x]**Storage 路径穿越防护**：`storage.service.ts` 的 `getFilePath()` 添加 `path.resolve` + 前缀校验
- [x]**SSE reader 安全释放**：`ai-gateway.service.ts` 的 `realChatStream()` finally 中先 `reader.cancel()` 再 `releaseLock()`

## 阶段三：代码质量提升（P2 — 长期优化）

> 目标：改善代码可维护性，消除技术债务。

### 3.1 消除 God Class 与重复代码

- [x]**plan-engine 模板外置**：6 个场景模板提取为独立 JSON 文件（`templates/sharing-prep.json` 等），`PlanEngine` 改为读取 JSON 模板
- [x]**getGoalWithMembershipCheck 提取**：创建 `GoalPermissionService` 或在 `GoalService` 中添加公共方法，供 `ExecutionService` 和 `CollaborationService` 复用
- [x]**status style map 统一**：前端 `STATUS_STYLE_MAP` 提取到 `lib/task-constants.ts`

### 3.2 类型安全加固

- [x]**execution.service.ts**：`Record<string, unknown>` 替换为 `Prisma.ExecutionTaskWhereInput` / `Prisma.ExecutionTaskUpdateInput`
- [x]**collaboration.service.ts**：同上，使用 Prisma 生成的类型
- [x]**plan-engine.ts**：`as` 断言替换为类型守卫或显式校验函数
- [x]**api.ts**：响应数据用 zod schema 校验后再 `as T`，或至少添加运行时类型检查

### 3.3 健壮性提升

- [x]**AI 调用重试**：`ai-gateway.service.ts` 的 `realChat()` 增加 429/5xx 重试逻辑（最多 3 次，指数退避）
- [x]**API 请求超时**：`api.ts` 的 `apiFetch()` 添加 `AbortSignal.timeout(15000)`
- [x]**monitoring 指标持久化**：`requestStats` Map 定期写入 DB 或至少输出到日志（防重启丢失）
- [x]**Session 清理 Job**：TaskWorker 新增 `SESSION_CLEANUP` 任务类型，定期 `DELETE FROM Session WHERE expiresAt < datetime('now')`

### 3.4 配置外置

- [x]**bcrypt salt rounds 可配置**：`auth.service.ts` 读取 `BCRYPT_SALT_ROUNDS` 环境变量（默认 12）
- [x]**monitoring 阈值可配置**：Disk 阈值、Job backlog 阈值、慢请求阈值从环境变量读取
- [x]**速率限制配置统一**：`rate-limit.middleware.ts` 的环境变量名与 `.env.example` 保持一致
