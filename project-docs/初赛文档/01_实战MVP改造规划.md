# 实战 MVP 改造规划

## 目标

当前项目已经具备可演示的网页 Demo 形态，页面完成度、技术栈和整体 UI/UX 风格都已形成稳定表达。接下来的目标不是重做页面，也不是扩展成复杂平台，而是在**不破坏现有技术栈与视觉风格**的前提下，把现有 Demo 升级为一个更接近真实使用场景的实战 MVP。

本次改造的核心方向是：

- 保留当前单页产品体验
- 保留当前技术栈与现有组件结构
- 保留当前视觉风格、文案气质与比赛展示感
- 强化输入信息的结构化程度
- 强化输出结果的可执行性
- 强化规划逻辑对不同场景约束的响应能力

## 约束

本项目后续改造必须遵守以下边界：

- 不更换现有技术栈
- 不随意推翻现有 UI/UX 风格
- 不把项目扩展成多场景平台
- 不擅自引入后端服务、数据库或登录系统
- 不擅自接入真实大模型 API
- 项目内容中不得写入与内部协作身份相关的私人称谓或信息
- 允许保留与 `TRAE`、比赛语境、Demo 创作过程相关的公开比赛线索

## 当前项目状态判断

从当前代码结构来看，现有项目已经具备以下基础：

- 具备稳定的单页展示结构
- 具备清晰的输入、处理中、结果展示闭环
- 已形成适合截图、录屏和比赛发帖的页面表达
- 已具备基础的规划逻辑与测试结构

但如果从“实战 MVP”角度看，当前版本仍主要偏向展示型 Demo，原因在于：

- 输入信息仍然偏粗，主要依赖一句自然语言目标
- 输出结果仍偏通用模板，个性化程度有限
- 规划逻辑对真实约束条件的响应还不够强
- 结果虽然可看，但离“可以直接照着执行”还差一层

因此，下一阶段最值得投入的，不是重新设计界面，而是升级规划核心，让项目从“好看的比赛 Demo”变成“更像真实可用的小产品”。

## 代码级改造清单

### 1. `src/lib/planner.js`

这是本次改造的核心。

当前逻辑以固定模板为主，仅做了少量关键词识别。下一步应升级为基于输入字段的轻量规划引擎，使结果能够根据不同约束生成不同侧重的行动方案。

改造方向：

- 从“固定模板填充”升级为“根据输入字段动态生成”
- 增加对听众对象、分享时长、分享日期、分享目标、当前准备状态的响应
- 根据不同约束调整阶段重点、行动顺序、提醒事项和结构建议
- 保持输出结果稳定、清晰、适合页面展示

### 2. `src/data/demoTemplates.js`

当前模板内容整体可用，但更偏展示型。下一步不需要推翻现有表达，而是补充更贴近真实筹备的模板素材，为规划逻辑提供更强支撑。

改造方向：

- 保留当前整体文案风格
- 增加不同分享场景下的行动建议素材
- 为不同准备阶段提供更细的任务颗粒度
- 为风险提醒、完成标准、结果建议提供更真实的模板基础

### 3. `src/App.jsx`

当前页面结构不需要推翻，应继续保留单页闭环体验。主要任务是增强状态管理和输入数据流，使页面能够承接更完整的输入结构和结果结构。

改造方向：

- 保持当前页面架构与信息节奏
- 让输入从单一目标文本扩展为“自然语言 + 关键字段”
- 保持现有加载与结果切换体验
- 不改变整体风格与页面骨架

### 4. `src/components/HeroSection.jsx`

输入区是本轮改造的重点之一，但不应破坏首屏表达力。

改造方向：

- 在现有视觉风格下增强输入能力
- 支持填写分享主题、听众对象、时长、日期、目标、当前准备状态等关键信息
- 保留当前输入区的第一视觉重点地位
- 保持组件风格统一，不改现有产品气质

### 5. `src/components/StageSection.jsx`

阶段区块当前已经具备展示能力，下一步应增强“执行感”。

改造方向：

- 保留当前卡片结构与阅读节奏
- 让每个阶段不仅有任务，还能体现任务目标与完成标准
- 强化阶段顺序的现实合理性
- 让评审和用户都能更直观感受到“这不是模板，是行动路径”

### 6. `src/components/ChecklistPanel.jsx`

这个区块非常适合承担“收口”作用，应进一步增强它的实战价值。

改造方向：

- 从普通清单升级为最终行动清单
- 突出“现在就可以开始做什么”
- 强化可执行性和完成感
- 保持整体样式不变，仅提升内容质量

### 7. `src/components/InsightPanel.jsx`

这一块适合作为“策略提醒区”，不必大改样式，但应让内容更贴近真实筹备风险。

改造方向：

- 保留当前双栏或卡片式提醒结构
- 强化亮点提示与风险提示的实用价值
- 让建议更像真实经验，而不是通用提示

### 8. `src/lib/planner.test.js`

当前测试已具备基础，但覆盖面仍偏窄。下一步应补齐不同输入条件下的行为验证。

改造方向：

- 增加不同听众对象下的输出差异测试
- 增加不同时长下的结果结构测试
- 增加不同目标和准备状态下的建议变化测试
- 保证规划逻辑升级后仍稳定可控

## MVP 功能优先级

### P0：必须优先完成

这些能力直接决定项目是否能从 Demo 升级为实战 MVP。

#### 1. 结构化输入

至少支持以下关键信息：

- 分享主题
- 听众对象
- 分享时长
- 分享日期
- 分享目标
- 当前准备状态

目标：让系统接收的不是一条模糊句子，而是一组能够驱动规划逻辑的真实约束。

#### 2. 动态结果生成

生成结果至少要覆盖：

- 分阶段筹备路径
- 每阶段任务清单
- 最终行动清单
- 风险提醒
- 分享结构建议
- 基于时间与准备状态的推进建议

目标：输出结果必须更像真实可执行计划，而不是统一模板。

#### 3. 单页闭环体验保持稳定

整体体验继续保持：

- 输入
- 生成
- 展示结果
- 形成收口

目标：保留比赛展示优势，不把产品做散。

#### 4. 比赛展示感不下降

必须继续保持：

- 页面适合截图
- 页面适合录屏
- 首屏表达清晰
- 结果区一眼能看懂

目标：即使进入实战 MVP 方向，也不能损失比赛展示效果。

### P1：在 P0 完成后继续增强

这些能力不是第一优先级，但完成后会明显提升产品可信度和实用性。

#### 1. 场景模板增强

支持针对不同分享场景给出不同参考起点，例如：

- 团队内部经验分享
- 新人培训分享
- 工具演示分享
- 复盘类分享

#### 2. 输出粒度随时长变化

根据分享时长不同，自动调整建议结构的细致程度。

例如：

- 10 分钟分享更强调轻量结构和节奏控制
- 30 分钟分享更强调案例组织和层次展开
- 60 分钟分享更强调章节设计和互动预留

#### 3. 根据准备状态调整建议起点

如果用户已经完成部分内容，系统应能从合适阶段开始推进，而不是一律从头开始。

#### 4. 增加更明确的完成标准

让每个阶段的任务不只是“要做什么”，还体现“做到什么程度算完成”。

#### 5. 增加可直接复用的结果区块

例如：

- 可直接照着执行的行动清单
- 可直接用于准备分享结构的结果摘要
- 可直接用于演讲前检查的风险排查清单

## 建议的实施顺序

为保证风险最小，建议按以下顺序推进：

1. 先改 `planner.js` 的输入与输出数据结构
2. 再补 `demoTemplates.js` 的内容支撑
3. 再调整 `HeroSection.jsx` 以承接结构化输入
4. 再增强 `StageSection.jsx`、`ChecklistPanel.jsx`、`InsightPanel.jsx` 的内容表达
5. 最后补齐 `planner.test.js`，保证逻辑稳定

## 结论

当前项目最值得做的，不是重写页面，也不是增加复杂技术能力，而是把现有规划逻辑升级为一个更像真实筹备工具的轻量引擎。

换句话说，本项目下一阶段的关键不是“让它看起来更厉害”，而是“让它真的更能帮用户推进一场分享会的准备工作”。

这也是把现有比赛 Demo 升级为实战 MVP 的最短路径。
