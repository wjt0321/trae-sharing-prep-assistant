# 真实数据实装与功能核验实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `subagent-driven-development` or `executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 为项目补齐三组真实业务数据，完成核心链路运行核对，修复阻断问题，并沉淀问题修复报告。

**Architecture:** 以后端种子数据扩展为主，补充少量可重复执行的接口核验脚本，再通过关键页面走查发现前后端联动问题。所有产物统一沉淀到 `project-docs/项目文档`，避免落到静态展示目录。

**Tech Stack:** Prisma + SQLite、NestJS 11、Next.js 16、TypeScript、PowerShell、项目内脚本

## 全局约束

- 所有新增项目文档统一放在 `d:\competition_notes\project-docs`
- 优先扩展 `apps/server/prisma/seed.ts`，必要时再拆分辅助数据文件
- 真实业务数据必须支持重复执行，不制造重复脏数据
- 输出文档避免出现“test”“测试”字样，统一使用“功能核验”“运行核对”“回归确认”“问题修复报告”
- 后端改动后执行 `pnpm --filter @ai-task-manager/server typecheck` 与 `pnpm --filter @ai-task-manager/server build`
- 前端改动后执行 `pnpm --filter @ai-task-manager/web typecheck` 与 `pnpm --filter @ai-task-manager/web build`
- 完成前必须形成 `project-docs/项目文档/13_真实数据功能核验与问题修复报告.md`

## Task 1：扩展真实业务数据

**Files:**

- Modify: `apps/server/prisma/seed.ts`
- Create or Modify: `apps/server/prisma/seed.real-scenarios.ts`

**Produces:**

- 3 组真实业务数据：比赛项目推进、团队分享筹备、内容创作发布
- 多工作区、多成员、多目标、多规划版本、多任务状态、多评论、多导出、多资产、多通知的幂等落库逻辑

- [ ] 梳理现有 `seed.ts` 中已创建的数据主键、邮箱、工作区 ID、目标 ID，列出不能冲突的唯一键
- [ ] 设计三组数据常量，统一命名前缀，例如 `real-competition-*`、`real-sharing-*`、`real-content-*`
- [ ] 先为每组数据建立用户和工作区，再补工作区成员关系与默认工作区映射
- [ ] 为每组数据补目标记录，并确保场景类型、约束、成功标准、协作标记完整
- [ ] 为关键目标补规划版本、任务、任务状态历史、评论、活动流、导出记录、资产、通知
- [ ] 将所有写入逻辑改为 `upsert` 或“先查后写”，保证重复执行稳定
- [ ] 执行 `pnpm --filter @ai-task-manager/server db:seed`
- [ ] 用 Prisma Studio 或接口抽查确认三组数据都能被读取

## Task 2：补充接口运行核对脚本

**Files:**

- Create: `apps/server/scripts/functional-check.mjs`
- Modify if needed: `apps/server/package.json`

**Consumes:**

- Task 1 生成的真实业务数据

**Produces:**

- 一套可重复执行的接口运行核对脚本
- 核对结果输出，覆盖认证、工作区、目标、规划、执行、协作、导出、通知、知识、后台链路

- [ ] 编写脚本基础设施：HTTP 请求封装、鉴权头注入、统一结果输出、失败即退出
- [ ] 先实现认证链路核对：登录、读取当前用户、读取工作区列表
- [ ] 实现主链路核对：目标列表、目标详情、规划版本、执行任务、协作评论、导出记录
- [ ] 实现侧链路核对：模板列表、资产列表、通知列表、审计统计、监控指标
- [ ] 若接口需要写操作，优先使用已有真实数据上的安全动作，如标记通知已读、读取共享页、生成一次新导出
- [ ] 将脚本命令加入 `apps/server/package.json`，例如 `pnpm functional:check`
- [ ] 在后端启动后执行脚本，保存首轮结果，用于定位实际问题

## Task 3：修复后端阻断问题

**Files:**

- Modify: `apps/server/src/modules/**`
- Modify if needed: `apps/server/src/presentation/**`

**Consumes:**

- Task 2 输出的接口异常与数据异常

**Produces:**

- 可通过接口运行核对的后端主链路

- [ ] 先按 P0/P1 优先级整理问题，例如查询失败、关系缺失、状态流转报错、共享链接异常
- [ ] 每修复一个问题后，优先复跑对应接口，再复跑全量脚本
- [ ] 如问题源于种子数据结构而非业务逻辑，优先修正种子数据，不做无必要重构
- [ ] 如问题源于 service/controller/dto 的校验或返回结构，保持局部修复，避免破坏既有功能
- [ ] 修复完成后执行 `pnpm --filter @ai-task-manager/server typecheck`
- [ ] 再执行 `pnpm --filter @ai-task-manager/server build`
- [ ] 记录每个问题的现象、根因、修复动作与回归结果，供报告沉淀
