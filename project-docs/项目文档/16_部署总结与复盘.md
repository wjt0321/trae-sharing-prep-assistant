## 问题与解决过程

### 问题 1：CI Failing after 5s

**现象**：推送后 GitHub Actions 两个 job 都在 5-6 秒内失败。

**排查**：
- 本地完整跑过 `install → build shared → typecheck → build → test` 全部通过
- 5 秒失败时间太短，排除代码问题
- 检查 `package.json` 发现 `"packageManager": "pnpm@10.28.0"`
- 检查 CI 工作流发现 `pnpm/action-setup@v4` 指定 `version: 10`（不完整 semver）

**根因**：`packageManager` 字段与 CI 显式 `version` 参数冲突，pnpm setup 步骤直接失败。

**修复**：移除 CI 里的 `version: 10`，让 action 自动读取 `package.json` 的 `packageManager` 字段。

**提交**：`263befb`

### 问题 2：CI Failing after 26s

**现象**：第一轮修复后 CI 跑到 26-34 秒仍失败。

**排查**：
- 用 `gh run view --log-failed` 拉取日志
- 发现 `tsc --noEmit` 报 60+ 个 `TS7006: Parameter implicitly has 'any' type`
- 错误集中在 `audit.service.ts`、`auth.service.ts`、`collaboration.service.ts` 等
- 本地 typecheck 通过，CI 失败 → 环境差异

**根因**：CI 全新 `pnpm install` 后 Prisma client 未生成。Prisma 查询结果类型为 `any`，导致 `.map((t) => ...)` 的 lambda 参数全部隐式 `any`，触发 `noImplicitAny` 报错。本地通过是因为之前跑过 `prisma generate`，client 类型已缓存在 `node_modules/.prisma/client`。

**修复**：CI 的两个 job 在 `Build shared package` 之后、`Typecheck`/`Test` 之前，加一步：

```yaml
- name: Generate Prisma client
  run: pnpm --filter @ai-task-manager/server exec prisma generate
```

**提交**：`e16fce8`

**验证**：CI 重新跑通，`conclusion: success`。

### 问题 3：GitHub Pages 还是旧 demo

**现象**：用户期望 Pages 显示新 app，实际仍显示 legacy-demo。

**排查**：
- `docs/index.html` 的 `<title>分享会筹备助手</title>` 就是 legacy-demo
- 新 monorepo 是 Next.js + NestJS 全栈应用，需要 Node.js 运行时 + 数据库
- GitHub Pages 只能托管静态文件

**结论**：这不是 bug，是架构限制。新 app 物理上无法部署到 GitHub Pages。

**决策**：采用 Option A —— Pages 继续放 legacy-demo 做门面，新 app 本地演示。同时生成 Vercel + Railway 部署指南供后续产品化使用。

### CI 流水线（.github/workflows/ci.yml）

**quality job**：
1. Checkout
2. Setup pnpm（读 packageManager 字段）
3. Setup Node.js 20（cache: pnpm）
4. `pnpm install --frozen-lockfile`
5. `pnpm --filter @ai-task-manager/shared build`
6. `pnpm --filter @ai-task-manager/server exec prisma generate` ← 关键步骤
7. `pnpm run typecheck`
8. `pnpm run lint`（continue-on-error）
9. `pnpm run build`

**backend-test job**：
1-6 同上
7. `pnpm --filter @ai-task-manager/server test`

### GitHub Pages

- **源**：`docs/` 目录（master 分支）
- **内容**：legacy-demo 构建产物（Vite 静态单页）
- **地址**：https://wjt0321.github.io/trae-sharing-prep-assistant/
- **更新方式**：`cd legacy-demo && npm run build`（产物输出到 `docs/`）

## 关键决策记录

### 决策 1：Pages 保留 legacy-demo（Option A）

**背景**：新 monorepo 无法部署到 GitHub Pages（需服务端运行时）。

**选项**：
- A. Pages 放 legacy-demo，新 app 本地演示
- B. 部署到 Vercel + Railway（需改 Prisma 为 PostgreSQL）
- C. Next.js 静态导出（功能全失，无意义）

**选择**：A（比赛阶段）+ B（赛后产品化）

**理由**：比赛演示用本地跑完整功能最可靠，Pages 做门面页持续在线。赛后走 B 路线做真实产品。

### 决策 2：CI 用 packageManager 字段而非显式 version

**背景**：`pnpm/action-setup@v4` 的 `version` 参数与 `package.json` 的 `packageManager` 字段冲突。

**选择**：移除 CI 的 `version` 参数，让 action 自动读取 `packageManager`。

**理由**：单一数据源，避免版本不一致。`package.json` 是 pnpm 版本的权威声明。

### 决策 3：CI 显式生成 Prisma client

**背景**：pnpm install 后 Prisma client 不自动生成，导致 typecheck 失败。

**选择**：CI 加独立 `prisma generate` 步骤。

**理由**：Prisma 的 postinstall 在 pnpm workspace 下不可靠，显式生成最稳妥。

## 复盘要点

### 做得好的

1. **问题定位高效**：CI 5s 失败 → 排除代码问题 → 聚焦环境配置；26s 失败 → 拉日志 → 精确定位 Prisma client
2. **本地复现验证**：所有 CI 命令本地跑通后再推送，减少试错成本
3. **文档同步**：部署指南、复盘文档同步产出，不积压
4. **决策有据**：Pages 策略基于架构限制而非主观偏好，保留升级路径

### 可改进的

1. **CI 应在 monorepo 骨架搭建时就跑通**：而不是等图标替换后才第一次验证 CI。后续新增模块时 CI 应作为合并门槛。
2. **Prisma generate 应纳入 dev workflow**：考虑在 `apps/server/package.json` 加 `postinstall: prisma generate`，避免本地新克隆仓库也踩坑。
3. **演示数据应更早填充**：用户反馈"看的多改的少"正是因为 seed 数据是完成态。后续应在开发阶段就维护演示数据。
4. **Pages 策略应提前沟通**：用户对 Pages 期望与新 app 能力有认知差，应在项目初期就明确 Pages 的定位和限制。

### 技术债

| 债务 | 严重度 | 说明 |
|---|---|---|
| Prisma 仍是 SQLite | 中 | 部署到 Railway 前必须切换 PostgreSQL |
| uploads/ 本地存储 | 中 | Railway 部署后文件不持久，需迁移对象存储 |
| AI 默认 mock 模式 | 低 | 比赛演示需手动配置真实模型 |
| legacy-demo 与 monorepo 并存 | 低 | 两套代码维护成本，赛后考虑归档 legacy-demo |
