# 功能迭代方向路线图

## 一、文档目的

本文件用于承接 `分享会筹备助手` 在 **实战 MVP 已完成** 之后的下一阶段功能演进判断。

它解决的问题不是：

- 比赛材料怎么收口
- 演示脚本怎么写
- 答辩口径怎么统一

而是更聚焦于：

- 当前产品在功能层面还薄弱在哪里
- 如果继续迭代，最值得增强的能力是什么
- 应该按什么顺序增强，才能既保住现有风格，又让产品更像真实工具

这份路线图默认采用 **方案 C：分阶段混合推进**。

也就是说，后续功能迭代不走“只补一点点”或“一次做成大平台”两种极端，而是按下面的顺序推进：

1. 先把单次生成结果做深
2. 再把生成后的持续推进能力做起来
3. 最后补齐场景模板与结果复用能力

---

## 二、当前产品判断

从当前版本来看，项目已经完成了 MVP 的基本闭环：

- 有结构化输入
- 有轻量规则引擎
- 有分阶段结果展示
- 有行动清单、结构建议和提醒信息
- 有稳定的单页演示体验

但如果从“真实可用的小产品”来看，当前仍有几个明显短板：

### 1. 单次生成仍然偏浅

虽然结果已经比最初 Demo 更完整，但很多建议仍停留在“方向正确”的层面，离“用户拿到后就能直接照着做”还有距离。

主要表现为：

- 阶段任务仍偏通用
- 不同场景间的差异还不够大
- 缺少更细的执行颗粒度
- 缺少对内容准备质量的进一步判断

### 2. 结果生成后就停止了

当前产品更像一个“生成一次方案”的工具，而不是一个“帮助用户持续推进准备过程”的助手。

主要表现为：

- 用户无法基于结果继续微调
- 无法重排优先级
- 无法记录当前完成进度
- 无法把准备过程拆成多轮推进

### 3. 场景深度和复用能力不够

当前项目已经支持部分输入变化，但还没有形成真正的“场景模板库”或“可复用成果区块”。

主要表现为：

- 不同类型分享的专用方案不够清楚
- 结果难以直接二次使用
- 缺少可复制、可导出、可继续整理的成果形态

---

## 三、迭代总原则

后续迭代必须继续遵守以下边界：

- 不更换当前技术栈
- 不引入后端、数据库、登录系统
- 不接入真实大模型 API
- 不把项目扩成多场景平台
- 不破坏现有 `Warm Editorial + Minimal Product` 风格
- 不让页面滑向后台系统、复杂表单工具或聊天机器人

功能增强的核心原则应统一为：

- 不是为了让功能变多，而是为了让用户更容易推进一场分享的准备
- 不是为了做复杂平台，而是为了让单场景产品更完整
- 不是为了炫技，而是为了让结果更可执行、更可延续、更可复用

---

## 四、推荐路线

推荐采用三阶段路线图推进：

### Phase 1：先把“单次生成”做深

这个阶段的目标，是让用户第一次点击生成之后，拿到的就不只是“像样的建议”，而是一份更接近真实执行方案的结果。

这一阶段最值得增强的能力包括：

#### 1. 更细的场景识别

在现有“听众对象、时长、目标、准备状态”基础上，继续识别更具体的分享类型，例如：

- 团队经验分享
- 新人培训
- 工具演示
- 项目复盘
- 方案汇报

意义：

- 让输出不只是因输入字段变化而变化
- 而是因“场景类型”变化而切换不同的筹备思路

#### 2. 更细的阶段任务颗粒度

当前四阶段框架可以保留，但每个阶段内部应提供更明确的执行动作，例如：

- 要准备哪些内容素材
- 要确认哪些现场条件
- 要做到什么程度算可进入下一阶段
- 哪些动作是必须做，哪些是可选加强

意义：

- 让阶段卡片从“展示型说明”变成“执行型路径”

#### 3. 更强的结果差异化

后续可以让不同输入条件影响更多结果区块，例如：

- 阶段重点不同
- 优先行动清单不同
- 风险项不同
- 分享结构建议不同
- 预演建议不同

意义：

- 强化“不是模板填空”的感受
- 提升结果可信度

#### 4. 增加内容质量自检项

当前系统更偏“准备过程规划”，后续可补一层“内容质量检查”，例如：

- 主题是否收敛
- 结构是否过满
- 案例是否匹配听众
- 结尾是否有明确收口

意义：

- 让产品从“准备步骤整理器”升级为“准备质量辅助器”

#### Phase 1 的完成标准

- 不同场景下的结果差异明显增强
- 每个阶段都更像可执行任务，而不是说明文字
- 用户看完结果后更容易直接开始做事
- 页面仍然保持当前展示感与简洁度

---

### Phase 2：再把“持续推进”能力做起来

这个阶段的目标，是让产品不只负责生成结果，还能承接“生成之后怎么办”。

它会让产品从一次性生成器，开始更像一个轻量助手。

这一阶段最值得增强的能力包括：

#### 1. 结果区块支持继续调整

用户在拿到方案后，应该可以做有限度的二次调整，例如：

- 重排某些行动项的优先级
- 标记哪些阶段已完成
- 根据实际准备进度更新建议起点
- 切换“更保守 / 更紧凑 / 更完整”的推进节奏

意义：

- 让结果从静态展示变成可继续推进的工作面

#### 2. 增加阶段进度追踪

可以给四阶段增加极轻量的推进状态，例如：

- 未开始
- 进行中
- 已完成
- 建议下一步

意义：

- 让用户更容易知道自己现在处在哪一步
- 也让结果区更有“助手陪你推进”的感觉

#### 3. 增加“下一步行动”收口模块

相比泛化清单，更值得增加一个明确区块：

- 如果你现在时间很紧，先做什么
- 如果你已经有初稿，下一步做什么
- 如果你只剩 1 天，最该保什么

意义：

- 强化产品对现实节奏的响应能力
- 让用户打开页面后马上知道当前最重要的动作

#### 4. 增加“重新生成但保留上下文”的轻量机制

当前重新生成更像一次全量刷新，后续可以支持：

- 保留已选场景与约束
- 只替换某个字段重新出结果
- 围绕当前结果做“增强版 / 压缩版 / 稳妥版”变体

意义：

- 让产品更适合真实试错过程
- 用户不需要每次都从头填写

#### Phase 2 的完成标准

- 用户可以围绕生成结果继续推进，而不是只能重新开始
- 页面交互仍然克制，不会变成复杂任务管理器
- 持续推进能力明显增强，但产品定位仍是“分享会筹备助手”

---

### Phase 3：最后补齐“场景模板 + 结果复用”能力

这个阶段的目标，是让产品逐步形成更专业的内容资产，而不仅是每次即时生成。

这一阶段最值得增强的能力包括：

#### 1. 建立场景模板库

为高频分享类型提供更明确的起点模板，例如：

- 新人培训模板
- 工具分享模板
- 项目复盘模板
- 经验沉淀模板
- 对外介绍模板

这些模板不是让用户跳过规划，而是给出更合理的默认起点。

意义：

- 降低输入门槛
- 提高结果专业感
- 增强不同场景的识别度

#### 2. 增加可复用结果区块

后续生成结果可以逐步沉淀为可直接继续使用的内容模块，例如：

- 可复制的分享大纲建议
- 可复制的预演检查清单
- 可复制的现场准备清单
- 可复制的开场 / 收尾提醒

意义：

- 让结果不只是“看完就结束”
- 而是可以被带到实际准备工作中继续使用

#### 3. 增加多版本结果输出

针对同一分享目标，后续可以提供有限的结果变体，例如：

- 精简版
- 标准版
- 稳妥版
- 冲刺版

意义：

- 让产品更适应不同时间压力与准备状态
- 让同一场分享的筹备方式更有弹性

#### 4. 增加“分享结构参考卡”

目前已有结构建议，后续可以进一步做成更可复用的小成果，例如：

- 开场怎么立主题
- 中段怎么铺案例
- 结尾怎么收观点
- 什么情况下需要互动环节

意义：

- 把“结构建议”从一句话升级成可参考的内容模块

#### Phase 3 的完成标准

- 产品对不同分享类型有更明确的专业表达
- 结果具备一定复用价值
- 用户不只得到计划，还能得到一部分实际可用素材

---

## 五、优先级判断

如果后续只能继续做一轮功能增强，优先顺序建议如下：

### P0：最值得先做

- 更细的场景识别
- 更细的阶段任务颗粒度
- 更强的结果差异化
- 增加“下一步行动”收口模块

原因：

- 这些能力最能直接提升“结果可执行性”
- 同时对现有单页结构最友好
- 风险低，收益高，最适合当前阶段

### P1：P0 稳定后再做

- 阶段进度追踪
- 结果区继续调整
- 重生成时保留上下文
- 可复制的结果区块

原因：

- 这部分会明显增强产品感
- 但交互复杂度也会上升，需要在 P0 稳定后再推进

### P2：后续再做

- 场景模板库
- 多版本结果输出
- 更完整的结构参考卡

原因：

- 这部分更偏“资产沉淀”
- 对产品可信度有帮助，但不是当前最短板

---

## 六、不建议现在做的方向

为了避免项目失焦，以下方向当前不建议推进：

- 做成聊天式连续对话产品
- 做成多场景通用筹备平台
- 做成复杂项目管理工具
- 引入账号体系、云端存储、多人协作
- 接入真实模型接口后追求“生成更多话”

原因很明确：

- 会明显超出当前比赛型单页产品边界
- 会破坏项目当前最大的优势：聚焦、克制、好展示
- 会让功能复杂度快速上升，但未必提升实际完成度

---

## 七、建议的下一轮落地顺序

如果后续真的进入下一轮开发，建议按下面顺序推进：

1. 先增强 `planner` 的场景判断与结果差异化
2. 再增强阶段卡片的任务目标、完成标准和下一步动作
3. 再增加结果区的“当前先做什么”模块
4. 再尝试加入轻量进度追踪
5. 最后补场景模板与可复用结果区块

这样做的好处是：

- 每一轮都能单独交付
- 每一轮都能明显增强功能价值
- 每一轮都不需要推翻当前 UI 结构

---

## 八、结论

`分享会筹备助手` 当前最合理的后续方向，不是简单继续堆功能，也不是停止在比赛交付层面不再演进，而是沿着下面这条路径继续增强：

- 第一步：让结果更深
- 第二步：让结果能继续推进
- 第三步：让结果能复用和沉淀

这条路线既能回应当前“功能性还比较薄弱”的问题，也能继续保住项目已经形成的产品边界和展示气质。

如果只用一句话概括后续方向，可以统一为：

> 在不扩平台、不改风格的前提下，把 `分享会筹备助手` 从“能生成筹备方案的 MVP”，继续升级成“能帮助用户持续推进分享准备过程的轻量产品”。
