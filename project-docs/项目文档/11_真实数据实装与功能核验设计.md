# 真实数据实装与功能核验设计

## 1. 背景

当前 `AI 任务管家` 的产品骨架、前后端模块和基础种子数据已经就位，但现有示例数据主要用于演示初始化，尚不足以覆盖：

- 多工作区、多角色、多成员协作场景
- 多目标、多规划版本、多任务状态流转
- 导出分享、知识沉淀、通知联动、审计与监控等侧链路
- 页面在真实业务密度下的渲染稳定性与边界表现

本次工作目标是把仓库从“有模块、有骨架”推进到“有真实业务数据、可跑完整链路、可发现并修正缺陷、可沉淀问题修复报告”的状态。

## 2. 目标

### 2.1 主要目标

- 为系统补充三组可真实演示的业务数据
- 以真实业务流贯通 14 个后端模块与关键前端页面
- 通过接口核验与页面走查发现影响使用的问题
- 修复高优先级问题并完成回归确认
- 生成一份面向项目沉淀的《问题修复报告》

### 2.2 非目标

- 不做与本轮无关的大规模架构重构
- 不追求覆盖全部极端边界条件或性能压测
- 不把验证范围扩展到生产部署、云资源、第三方真实外部通道

## 3. 默认假设

用户已明确要求直接开始，因此本方案基于以下默认假设推进：

- 以现有 SQLite 本地环境为唯一数据源
- 以当前 monorepo 结构为实施边界，不拆分独立子项目
- 允许扩展 `prisma/seed.ts` 或新增数据准备脚本，但应优先复用现有种子机制
- 允许补充少量自动化校验代码与脚本，但以高价值回归点为主，不增加噪音
- 报告正文避免使用“test”“测试”等字样，统一采用“功能核验”“运行核对”“回归确认”“问题修复报告”等表述

## 4. 业务数据设计

### 4.1 数据包结构

本轮补充三组真实业务数据包，每组都不是孤立记录，而是覆盖“工作区 -> 成员 -> 目标 -> 规划 -> 执行 -> 协作 -> 导出 -> 资产 -> 通知”的链路组合。

### 4.2 数据包 A：比赛项目推进

定位：覆盖最完整主链路，作为答辩和产品演示主案例。

建议数据内容：

- 工作区：`TRAE 创造力大赛冲刺组`
- 成员：发起人、产品协作者、视觉协作者、内容协作者
- 目标：比赛立项、功能收口、答辩材料整理
- 规划：至少 2 个版本，体现首次规划与约束变化后的重规划
- 执行：覆盖 `pending / in_progress / completed / blocked / skipped / cancelled`
- 协作：评论、回复、解决、任务指派、活动流
- 导出：行动清单、汇报摘要、演示提纲、共享链接
- 资产：从导出沉淀案例资产，从目标沉淀团队模板
- 通知：评论提及、回复、任务指派、任务受阻、重规划提醒
- 审计与监控：登录、配置修改、导出分享、页面请求等记录可观察

### 4.3 数据包 B：团队分享筹备

定位：覆盖分享准备业务场景，强化目标创建、模板套用、执行推进与导出产物。

建议数据内容：

- 工作区：`产品分享筹备组`
- 目标：新人培训分享、季度经验复盘会
- 模板：官方场景模板 + 团队沉淀模板各一份
- 规划：强调阶段拆解、里程碑、风险与演讲素材准备
- 执行：以演讲稿、案例整理、彩排任务为主
- 导出：阶段计划、演示提纲、复盘摘要
- 资产：检查清单、洞察、案例类知识片段

### 4.4 数据包 C：内容创作发布

定位：覆盖内容规划、知识资产、复用与导出沉淀链路。

建议数据内容：

- 工作区：`内容增长工作区`
- 目标：系列文章发布、短视频脚本制作
- 规划：选题调研、提纲撰写、素材整理、发布节奏
- 执行：带有负责人、阻塞说明、最近更新时间
- 导出：行动清单、发布复盘
- 资产：从导出转案例、手工新建 insight/checklist/asset

## 5. 模块覆盖矩阵

### 5.1 必过模块

- `auth`：登录、刷新、资料修改、密码修改
- `workspace`：工作区创建、成员关系、角色权限、切换
- `goal`：目标创建、编辑、场景识别、模板预填
- `planning`：规划生成、重规划、版本切换、差异比较
- `execution`：任务同步、状态变更、批量更新、阻塞处理、进度统计
- `collaboration`：评论、回复、解决、活动流、任务指派
- `export`：导出生成、共享页、下载开关、资产沉淀
- `knowledge`：模板查询/应用、资产创建/筛选/删除
- `notification`：列表、未读数、已读、删除、事件联动
- `ai-config`：配置读取、保存、清除、连接校验、成本统计
- `prompt-registry`：模板列表、创建版本、激活版本、预览渲染
- `audit`：日志列表、筛选、聚合统计
- `monitoring`：健康检查、系统指标、业务指标

### 5.2 基础设施模块

- `ai-gateway`：mock/真实配置回退链路可运行
- `task-worker`：任务积压与处理状态可观察
- `storage`：导出文件保存、读取、删除可用
- `presentation`：统一响应包装、异常处理、速率限制不影响主流程

## 6. 核验设计

### 6.1 分层策略

采用三层核验方式，避免只看接口或只看页面：

1. 数据准备层：确认真实业务数据能够稳定写入，并且可重复执行
2. 接口链路层：以脚本或自动化调用贯通关键 API，确认模块联动正确
3. 页面走查层：在真实数据下打开关键页面，确认加载、空态、错误态、交互和联动结果

### 6.2 接口链路重点

- 认证链路：注册、登录、刷新、修改资料、修改密码
- 工作区链路：新建团队工作区、邀请成员、切换角色、拉取成员列表
- 目标链路：从空白创建、从模板创建、更新目标、场景识别
- 规划链路：生成规划、查看版本、重规划、切换激活版本、比较版本
- 执行链路：同步任务、单任务更新、批量状态调整、读取时间线
- 协作链路：评论、回复、解决、任务指派、活动流查询
- 导出链路：创建导出、更新共享设置、查看共享页、沉淀资产
- 通知链路：读取未读数、标记已读、删除消息
- 知识链路：模板筛选、模板应用、资产 CRUD、标签搜索
- 后台链路：AI 配置、提示词管理、审计统计、业务指标、系统指标

### 6.3 页面走查重点

- 登录页、注册页
- 首页与工作区总览
- 目标创建页、目标详情页
- 规划页、执行工作台、协作页、导出页
- 模板库页、资产库页、通知中心页
- 设置页各子页面
- 监控后台页
- 公开共享页

页面走查中重点关注：

- 数据加载时是否出现空白区域或崩溃
- 长文本、空列表、多个版本、多通知情况下的布局是否稳定
- 跨页跳转和操作回流是否正确
- 错误提示是否清晰、状态是否一致

## 7. 缺陷分级与处理原则

### 7.1 分级

- P0：主链路不可用，例如无法登录、无法创建目标、无法生成规划、页面崩溃
- P1：重要能力异常，例如任务状态流转错误、评论无法回复、共享页打不开
- P2：体验或边界异常，例如数据显示不完整、筛选条件失效、提示文案不准确
- P3：轻微表现问题，例如排版错位、局部文案不统一

### 7.2 处理策略

- P0 / P1：本轮必须修复并完成回归确认
- P2：优先修复影响演示或连续操作的问题
- P3：在时间允许时顺手修复，否则进入报告中的遗留项

## 8. 实施方案

### 8.1 数据实装

优先方案：

- 扩展 `apps/server/prisma/seed.ts`
- 增加结构化数据常量，按“用户/工作区/目标/规划/任务/评论/导出/资产/通知”分段落库
- 保证幂等执行，重复运行不会制造脏数据或重复关系

如 `seed.ts` 复杂度过高，则拆分为：

- `prisma/seed.core.ts`：基础账号与默认数据
- `prisma/seed.real-scenarios.ts`：多场景真实业务数据

### 8.2 自动化核验

建议补充一组高价值脚本，避免全靠手工点击：

- 认证与主链路脚本：登录、工作区、目标、规划、执行
- 协作与导出脚本：评论、指派、通知、导出、共享页、资产沉淀
- 后台链路脚本：AI 配置、提示词、审计、监控

实现形式优先级：

1. 复用后端现有 e2e 能力或新增面向 API 的场景脚本
2. 必要时补充轻量页面自动化走查
3. 对难以自动化但高风险页面采用人工浏览器核对

### 8.3 修复节奏

采用“小步快跑”节奏：

1. 先补数据
2. 跑接口链路
3. 修复阻断问题
4. 跑页面走查
5. 修复展示与联动问题
6. 进行全量回归确认
7. 产出问题修复报告

## 9. 报告设计

最终输出文档建议放在：

- `project-docs/项目文档/11_真实数据功能核验与问题修复报告.md`

报告结构：

- 背景与目标
- 本次补充的真实业务数据概览
- 功能核验范围
- 发现的问题列表
- 每个问题的现象、原因、修复动作、回归结果
- 未处理风险与后续建议
- 附录：关键截图、关键接口、关键页面列表

文案规范：

- 标题使用“功能核验”“运行核对”“问题修复报告”
- 避免出现“test”“测试”字样

## 10. 风险与回退

### 10.1 风险

- 现有种子数据与新增真实数据可能发生唯一键冲突
- 多模块联动后，幂等逻辑可能暴露重复写入问题
- 页面在高密度真实数据下可能出现意料外渲染问题
- 真实 AI 配置相关接口可能受环境变量或外部网络影响

### 10.2 回退策略

- 所有数据写入都应使用可重复执行的 `upsert` 或受控清理
- 对风险较高的修复优先保持局部改动，避免连带破坏主链路
- 若真实 AI 调用不可达，则保留 mock 回退，不阻断主流程核验

## 11. 验收标准

- 三组真实业务数据可稳定落库并被前端正确消费
- 关键页面在真实数据下可打开、可操作、无明显阻断
- 关键接口链路贯通，跨模块联动结果正确
- P0 / P1 级问题全部修复并完成回归确认
- 形成一份可供答辩与后续维护使用的问题修复报告

## 12. 下一步实施规划

实施阶段将拆为以下工作包：

1. 真实业务数据扩展
2. 接口链路脚本与回归点补齐
3. 页面走查与联动问题修复
4. 全量回归确认
5. 问题修复报告沉淀
