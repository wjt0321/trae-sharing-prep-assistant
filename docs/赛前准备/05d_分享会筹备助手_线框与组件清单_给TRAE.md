# 分享会筹备助手｜线框与组件清单（给 TRAE）

## 1. 这份文件的用途
这份文件是给 TRAE 的最后一层落地说明。

如果 `05` 是产品 PRD，`05b` 是执行提示词，`05c` 是视觉方向，
那么这份 `05d` 就是：

- 页面怎么排
- 每块放什么
- 用什么组件
- 哪些状态必须做
- 哪些地方最值得截图

目标是让 TRAE 不需要再猜布局，直接开始搭前端页面。

---

## 2. 页面整体线框
推荐做成一个单页，整体宽度控制在 `1120px` 左右，桌面优先。

### 页面结构
```text
┌──────────────────────────────────────────────┐
│ Header                                       │
├──────────────────────────────────────────────┤
│ Hero / 输入区                                │
│ 标题 + 副标题 + 输入框 + 示例 + 按钮         │
├──────────────────────────────────────────────┤
│ 结果区（上）                                 │
│ 4 个阶段任务卡片                             │
├──────────────────────────────────────────────┤
│ 结果区（下）                                 │
│ 左：最终行动清单  ｜ 右：准备重点 / 风险提醒  │
├──────────────────────────────────────────────┤
│ Footer / 轻说明                              │
└──────────────────────────────────────────────┘
```

---

## 3. 页面模块拆解

### 模块 A：Header
#### 作用
让页面像一个真实产品，而不是一个 demo 截图页。

#### 内容
- 左侧：产品名 `分享会筹备助手`
- 右侧：轻状态标签，例如 `Demo` / `TRAE 初赛作品`

#### 组件
- `AppHeader`
- `Badge`

#### 样式要求
- 高度轻
- 不抢视觉
- 边框很淡
- 不要复杂导航

---

### 模块 B：Hero 输入区
#### 作用
这是页面最重要的第一屏，也是首张截图重点。

#### 内容
- 主标题
- 副标题
- 多行输入框
- 主按钮 `开始拆解`
- 1-2 个示例输入 chip

#### 推荐文案
标题：
`把分享准备变成行动清单`

副标题：
`输入一个分享目标，系统会帮你拆解出准备步骤、执行顺序和最终行动清单。`

示例：
- `我要准备一场小型分享会，但我不知道该怎么安排。`
- `我要做一次 10 分钟的主题分享。`

#### 组件
- `HeroSection`
- `Textarea`
- `PrimaryButton`
- `ExampleChip`

#### 样式要求
- Hero 整体放进一张大卡片
- 标题明显但不要浮夸
- 输入框是视觉中心
- 按钮突出但不刺眼

---

### 模块 C：处理中状态
#### 作用
避免用户点击后只看到空白。

#### 内容
- 一句状态提示：`正在拆解准备步骤...`
- 轻量 skeleton 卡片 2-4 个

#### 组件
- `LoadingState`
- `SkeletonCard`

#### 样式要求
- 不要大转圈居中占满页面
- 要让用户感觉“结果马上要出来了”

---

### 模块 D：阶段任务卡片区
#### 作用
这是页面第二个重点，也是 Demo 的核心价值展示区。

#### 推荐阶段
1. 目标与主题确认
2. 内容准备
3. 时间与设备安排
4. 预演与收尾检查

#### 每张卡片内容
- 阶段编号或阶段标签
- 阶段标题
- 一句阶段说明
- 2-3 条具体任务

#### 示例结构
```text
阶段 1｜目标与主题确认
先明确这次分享的主题、受众和时间限制。
- 确定分享主题和目标
- 明确受众是谁
- 确定分享时长
```

#### 组件
- `StageSection`
- `StageCard`
- `StageTag`
- `TaskItem`

#### 布局建议
- 桌面端：2 列网格
- 移动端：1 列堆叠

#### 样式要求
- 卡片不要太挤
- 标题层级清楚
- 每条任务短、可执行
- 不要像后台列表

---

### 模块 E：最终行动清单区
#### 作用
这是第三个截图重点，体现“已经帮用户收口”。

#### 内容
- 一份最终行动清单
- 按 1、2、3、4… 展示

#### 示例内容
- 先确定主题与听众
- 再准备主要内容结构
- 提前确认时间安排与设备
- 分享前完成一次预演

#### 组件
- `ChecklistPanel`
- `ChecklistItem`

#### 样式要求
- 比阶段卡片更收口
- 可略加强调
- 看起来像真正可以照着做的结果

---

### 模块 F：准备重点 / 风险提醒区
#### 作用
让结果更完整，也更像真实产品建议。

#### 内容
建议拆成两个小块：
- `准备重点`
- `容易遗漏的事`

#### 示例内容
准备重点：
- 优先明确主题，不要一开始就写满内容
- 先搭结构，再补细节

容易遗漏的事：
- 忘记检查投屏与设备
- 没预留问答或缓冲时间

#### 组件
- `InsightPanel`
- `NoticeList`

#### 样式要求
- 不要太抢
- 作为右侧辅助区最合适

---

### 模块 G：底部轻说明
#### 作用
补一点产品身份信息即可。

#### 内容
- 一句小说明
- 一句 `Built with TRAE` 或类似字样

#### 组件
- `FooterNote`

#### 样式要求
- 很轻
- 不要复杂 footer

---

---

## 5. 页面状态清单
必须至少支持 3 个页面状态：

### 状态 1：初始态
- 有标题
- 有输入框
- 有示例
- 结果区为空态或占位说明

### 状态 2：处理中
- 显示处理中提示
- 显示 skeleton

### 状态 3：输出态
- 4 张阶段卡片
- 行动清单
- 重点 / 风险提醒

如果时间够，再补：

### 状态 4：再次输入态
- 用户修改输入后重新生成

## 7. 技术实现约束
请尽量按以下方式做：

### 推荐栈
- React
- Tailwind CSS
- 可选：`shadcn/ui`
- 图标：`lucide-react`

### 约束
- 不要上复杂全局状态管理
- 不要做复杂路由
- 不要做后端优先架构
- 不要因为组件库默认样式就偷懒
- 优先把视觉和结构做好

### 数据处理
如果接真实 AI 太慢，可以先本地 mock 出一套稳定结果，先保证页面演示质量。

---

## 8. 给 TRAE 的最终页面实现指令
请按以下优先级实现：

1. 先把单页结构搭出来
2. 先把 Hero 输入区做漂亮
3. 再把阶段任务卡片做漂亮
4. 再把行动清单和提醒区收口
5. 最后补 loading 状态和细节动画

永远记住：
- 这不是聊天页
- 这不是后台
- 这是一个适合比赛演示的任务拆解工作台

---

## 9. 最终验收标准
只要前端满足这些，就算真的可以交：
- 看起来像真实产品
- 不土、不乱、不像 AI 模板
- 输入区吸引人
- 阶段卡片清楚
- 行动清单有结果感
- 整页适合截图、录屏、比赛发帖
